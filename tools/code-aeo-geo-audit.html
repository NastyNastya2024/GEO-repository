Аудит кода под AEO и GEO (2026)

Рамка для ревью API, бэкенда и библиотек: насколько код отдаёт готовые ответы (логика AEO) и насколько он предсказуем и документирован для LLM и агентов (логика GEO).

Какой вопрос закрывает этот чеклист?

Контент‑страницы оптимизируют под сниппеты и нейросети отдельно (см. AEO и GEO для сайтов). Здесь — зеркальные принципы для исходного кода: чтобы downstream‑системы (поиск по коду, RAG над репозиторием, codegen, агенты) получали ясный контракт и «ответ», а не только сырые данные.

Метрики AEO для кода

AEO в коде — это способность сервиса выдавать точный, структурированный, сразу используемый результат (как «featured snippet» на уровне API), а не заставлять потребителя самому собирать смысл из внутренних полей.

Критерий Что проверить Сигнал «пройдено»
Прямой ответ Есть ли слой, который возвращает готовую формулировку/итог для вопроса пользователя (не только ORM‑строку или «сырой» JSON)? Отдельная функция или поле ответа с явным summary / answer / message для UI и ассистентов.
Структурированный вывод Для публичных ответов используются ли схемы (JSON Schema, OpenAPI components, типы DTO), при необходимости — связка с Schema.org на границе HTML? Контракт ответа описан; поля стабильны; версия схемы указана.
Инкапсуляция контекста Перед «тяжёлым» блоком есть ли краткое резюме (комментарий‑врезка, docstring с @returns смыслом, не только типом)? Читатель (и LLM) понимает назначение файла/модуля без чтения всей реализации.
Задержка ответа Для интерактивных сценариев (чат, голос) укладывается ли горячий путь в целевой бюджет (ориентир 100–200 мс на простой запрос или явно задокументировано иначе)? Есть измерения/профилирование; узкие места помечены; таймауты и деградация описаны.
Некорректный ввод Что возвращается при частичных/битых данных: осмысленный fallback, код ошибки, подсказка для клиента? Не «пустой 500»: структурированная ошибка + безопасное сообщение пользователю.

Метрики GEO для кода

GEO для кода — готовность к индексации и использованию генеративными системами: предсказуемые имена, явные контракты, объяснения «зачем», а не только «что делает строка».

Критерий Что проверить Сигнал «пройдено»
Комментарии «почему» Есть ли пояснения причин, инвариантов и компромиссов (не дублирование названия функции)? Новый разработчик или агент понимает контекст решения.
Сигнатуры и типы TypeScript, аннотации Python, generics там, где иначе неочевидно? Публичные API типизированы; unknown / any не маскируют контракт без причины.
Семантика имён Отражают ли имена сущностей предметную область (удобно для embedding и поиска по коду)? Нет «магических» data, res, tmp в публичных слоях без пояснения.
Описание API OpenAPI/Swagger, JSDoc, pydoc или аналог для входных/выходных форматов? Можно сгенерировать клиент или промпт по документации без угадывания.
Идемпотентность и стабильность Одинаковый валидный ввод даёт предсказуемый результат (детерминизм или явная документация недетерминизма)? Повтор запроса безопасен там, где заявлено; порядок полей в JSON не «плавает» без версии.
Подсказки для агентов Есть ли явные инструкции: «эта функция вызывается из…», «не вызывать без…», «side effects»? README/ARCHITECTURE или заголовочные комментарии задают правила игры.

Общие требования к коду

Критерий Что проверить
Без «зашитых» ответов там, где нужна шкала Тексты и правила вынесены в конфиг/i18n/БД; нет дублирования несогласованных формулировок в десятке мест.
Тесты на формат Юнит‑ или контракт‑тесты проверяют не только ветвление, но и форму ответа (snapshot/schema validation).
Ошибки как данные Ошибки сериализуются в структуру (код, поле, сообщение для человека, детали для логов), а не только throw без контекста на границе API.
Модульность слоёв Разделение: доступ к данным → бизнес‑логика → формирование ответа/DTO; нет смешения в одном «божественном» обработчике без причины.

Обязательная структура отчёта проверки

Итог аудита оформляйте строго в таком порядке — так проще сравнивать итерации и отдавать результат людям или LLM.

  1. Общая оценка AEO (шкала 1–10) + одна ключевая проблема (например: «нет слоя “answer”, только сырые сущности»).
  2. Общая оценка GEO (шкала 1–10) + одна ключевая проблема (например: «публичные функции без типов и OpenAPI»).
  3. Список нарушений по каждому пункту чеклистов выше: формулировка риска + короткая цитата или ссылка на файл:строка (как доказательство).
  4. Рекомендации по исправлению: конкретные шаги (рефакторинг, новый DTO, тест на schema), при желании — пример патча или псевдокод.
  5. Итоговый вывод: готов ли код к использованию в поисковых/генеративных контурах (да/частично/нет) и что блокирует продакшен.

Как фиксировать нарушение с цитатой из кода

В отчёте указывайте путь и фрагмент, который демонстрирует проблему. Пример типичного нарушения AEO: ответ без явного «итога» для потребителя. 

// Нарушение: возвращается внутренняя структура без поля ответа
function getOrderStatus(orderId) {
  return db.orders.findById(orderId);
}

Рекомендация: добавить слой toPublicOrderStatus(row) с полями answer, state, nextSteps и схемой JSON Schema для ответа API.

Проведи аудит приложенного кода по чеклистам:
- AEO: прямой ответ, структура вывода, контекст/резюме, задержка, fallback при битом вводе.
- GEO: комментарии «почему», типы, семантика имён, описание API, предсказуемость, подсказки для агентов.
- Общие: нет лишних hardcode-текстов, тесты формата, структурированные ошибки, слои данные→логика→ответ.

Выведи: (1) оценка AEO 1–10 + главная проблема, (2) оценка GEO 1–10 + главная проблема,
(3) нарушения по пунктам с цитатами из кода, (4) рекомендации, (5) итог «готов / не готов».
